## Bugs Found & Fixed

### Bug #1 — `maxRetries` uses `||` instead of `??` nullish coalescing

**File:** `cli/lib/ralph-inner.mjs:149`

**Problem:** `const maxRetries = config.maxRetries || 3;` — JavaScript's `||` treats `0` as falsy, so `maxRetries: 0` was silently overridden to `3`. This meant disabling retries by setting `maxRetries: 0` was impossible. Any config with `maxRetries: 0` would behave as `maxRetries: 3`.

**Symptoms:** `runPhase` never escalated when `maxRetries: 0` was set. The escalation check `retryCount >= maxRetries` would evaluate `retryCount >= 3` instead of `retryCount >= 0`.

**Fix:** Changed to `config.maxRetries ?? 3` (nullish coalescing). Now `0` is preserved, and only `undefined`/`null` fall back to `3`.

**Impact:** Low — the default `maxRetries` is `3` in `getDefaultConfig()`. This bug only affected explicit `maxRetries: 0` configurations. On fix, `maxRetries: 0` now correctly escalates on the first same-phase re-run (retryCount starts at 0, transitionPhase increments to 1, 1 >= 0 triggers escalation).

### Bug #2 (SPEC GAP) — SIMPLIFY phase uses generic feature-iterate template

**File:** `cli/lib/ralph-inner.mjs` — `buildFeatureIterateOutput()` lines 288-315

**Problem:** The SIMPLIFY phase used the same generic feature-iterate template as BUILD and VERIFY. The spec (PROJECT_PLAN.md lines 602-631) requires SIMPLIFY-specific instructions with:
- Planner: identify code smells, excessive nesting, DRY violations, premature optimization
- Simplifier (Generator persona): refactor for clarity with specific rules (max 4 nesting levels, no dead code, etc.)
- Evaluator: verify against simplify-specific criteria (dead code, nesting, DRY, test preservation)

**Symptoms:** Running `harness-dev phase simplify` printed BUILD/VERIFY-style instructions instead of SIMPLIFY-specific guidance. The agent would not know the SIMPLIFY scope, criteria, or constraints.

**Fix:** Added `buildSimplifyOutput()` function with full spec-compliant SIMPLIFY instructions. Modified `buildFeatureIterateOutput()` to dispatch to `buildSimplifyOutput()` when `phase === 'simplify'`.

**Verification:** `test-t10.mjs` suite C verifies all spec-required content is present in SIMPLIFY instructions: code smells, nesting, DRY, dead code, test preservation, and `--feature` flag reference.

### Identified Gaps

#### GAP #1 (MINOR) — `continuePipeline` autopilot path doesn't check pause

**Problem:** `continuePipeline` in autopilot mode transitions to the next phase without re-checking `config.paused`. The phase.mjs entry point checks pause before calling `transitionPhase`, but once the pipeline is running, the auto-advance doesn't re-verify.

**Impact:** LOW — In practice, the user cannot inject a pause between `runPhase` completing and `continuePipeline` running because both happen synchronously in the same command. The pause check at phase.mjs entry handles the "don't start a new run while paused" case. The `config.paused = false` clear in `transitionPhase` means any pause is reset once a new phase starts.

**Recommendation:** Add a pause check in `continuePipeline`'s autopilot path before the recursive call to `transitionPhase`. Defer to T12 if priority is low.

#### GAP #2 (MINOR) — `continuePipeline` bypasses escalation in direct call

**Problem:** Calling `continuePipeline(dir, 'build', ...)` directly from an escalated state transitions to the next phase (verify), which resets retryCount to 0. The escalation is lost. This only affects direct API callers — the normal `phase.mjs` flow only calls `continuePipeline` when `runPhase` returns `status: 'complete'`.

**Impact:** LOW — The recommended API usage pattern (through `phase.mjs`) always handles escalation correctly. Document this as a caveat for direct API users.

**Recommendation:** Add a pre-transition check in `continuePipeline` autopilot path: verify the current phase hasn't exceeded maxRetries before transitioning.

#### GAP #3 (DEFERRED) — `resume` command is a stub

**File:** `cli/commands/resume.mjs` — Returns `status: 'not_implemented'`. The resume functionality belongs to T12 (Autopilot Mode). Not a T10 gap.

#### GAP #4 (PRE-EXISTING) — T5 test G.5: `get()` returns null instead of undefined

**File:** `cli/lib/state.mjs` `get()` function — the `getKey()` helper returns `null` for missing keys, but the test expects `undefined`. Pre-existing issue (T5 baseline).

#### GAP #5 (PRE-EXISTING) — T6 test H.3: empty fields display inconsistency

**File:** `cli/lib/progress.mjs` — Empty fields print `""` instead of `"—"`. Pre-existing issue (T6 baseline).

#### GAP #6 — `cli/commands/validate.mjs` references `--feature`/`--task` flags but gates.mjs doesn't action them (KNOWN)

This is documented as a deferred gap since T7. It doesn't affect T10 correctness — the feature/task flags are parsed but not acted upon by the gate engine.
