### Identified Gaps

#### GAP #1 (CRITICAL) — `harness-dev validate --feature <name> --task <id>` NOT IMPLEMENTED

**Spec says:** Lines 386, 416, 436, 439 — `harness-dev validate --feature X --task Y` for per-task validation

**Reality:** `validate.mjs` does not parse `--feature` or `--task`. The args.mjs parser stores them as generic flags but `validate.mjs`'s `validateCommand()` function never reads `args.flags.feature` or `args.flags.task`. It always calls `runChecks(targetDir, phase)` which runs the full phase-level gate, not a per-task/subset check.

**Impact:** When `ralph-inner.mjs` tells the agent `Run: harness-dev validate --feature f1 --task t1`, the agent gets a full BUILD/VERIFY/SIMPLIFY gate result instead of a per-task check. This breaks validation granularity but doesn't crash — the command still runs and produces valid (phase-level) output.

**History:** This was flagged in the T7 audit as GAP #1 with the note "should be added when T8 (Ralph Inner Loop) is implemented" (see `references/t7-audit-2026-06-19.md`). T8 did not implement it.

**Fix requires:**
1. `validate.mjs` — parse `args.flags.feature` and `args.flags.task`
2. `gates.mjs` — add per-feature/task check functions or a filtering mechanism
3. When `--feature`/`--task` is provided, run only checks relevant to that task (e.g., lint that specific file, test that specific module)

#### GAP #2 (MINOR) — Config default `maxRetries: 3` vs Spec "Max iterations = 10 per phase"

**Spec says** (line 478): "Loop guard: Max iterations = 10 per phase (configurable)."

**Reality:** The `getDefaultConfig()` in `state.mjs` sets `maxRetries: 3`, and the JSON schema sets the default to `3`.

**Impact:** The engine works correctly with 3; this is just a documentation mismatch. 3 retries is actually more conservative and aligned with Hermes/Honcho patterns. Either update the spec to say 3, or update the default to 10.

#### GAP #3 (DESIGN NOTE) — Feature summary counts in config vs feature_list.json

The `harness-config.json` has a `features` object (`{remaining, passing, total}`) that tracks summary counts. The `feature_list.json` has detailed per-feature/per-task state. When `ralph-inner.mjs` marks features as passing (in the recursive case), it only updates `feature_list.json` via `saveFeatureList()` — it does NOT update the config's `features` summary.

**Impact:** `harness-dev status --json` shows `features: {remaining: 0, passing: 0, total: 0}` even when feature_list.json says 1/2 features pass. This is a cosmetic inconsistency. The agent workflow should update `harness-dev config set features.passing <n>` when features complete.

#### GAP #4 (DESIGN NOTE) — Iteration tracking is always 1

`runPhase()` always returns `iteration: 1`. There is no state persistence for iteration counts across retries. The T8 spec algorithm describes retry counting ("If fails (≤ maxRetries times) ... If fails > maxRetries times → escalate"), but this logic is deferred to the outer loop (T9) or the consuming agent.

**Impact:** Fine for T8 scope — T8 is a single-shot instruction generator. But T9 must integrate iteration tracking for retry behavior to work.

## Verdict

**T8 is ready for follow-on tasks (T9: Outer Ralph Loop, T10: Phase Orchestrator) with one must-fix recommendation:**

The `--feature`/`--task` flags on `harness-dev validate` **must be implemented before T9** — without them, the feature-iterate loop cannot do per-task validation, and any agent following the instructions will run full phase gates instead of task-specific checks. The implementation scope is contained to `validate.mjs` (flag parsing) and `gates.mjs` (per-feature/task check logic or passthrough).

The core `ralph-inner.mjs` module is **structurally sound** — it correctly implements the dual-loop Ralph pattern, the instruction-based agent model, and the feature list lifecycle. The 101/102 test score reflects the core implementation's quality.

**T8 Verdict: SOLID CORE, PENDING GATE** (gate verb: implement --feature/--task in validate before T9)
