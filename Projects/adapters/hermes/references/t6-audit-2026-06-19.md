# T6 Audit — progress.md Dual-Structure Writer (2026-06-19)

## Summary

Audited `cli/lib/progress.mjs`, `cli/commands/learn.mjs`, `cli/commands/status.mjs`,
and `templates/progress.md` for the T6 progress.md dual-structure writer implementation.

**Results:** 58/58 tests pass. Two bugs found and fixed (readProgress section
boundary detection, appendLesson file corruption). Test battery (58 tests) added
as `test-t6.mjs`.

## Bugs Found & Fixed

### Bug 1: readProgress else-if chain prevents sessionEnd detection

**File:** `cli/lib/progress.mjs`, `readProgress()` function, boundary detection loop

**Root cause:** The loop used `if/else if` for four conditions. When `## Lessons`
header was found (condition 2), the session-end boundary check (condition 3) was
never evaluated because `else if` short-circuits.

```javascript
// BROKEN: else-if chain
for (let i = 0; i < lines.length; i++) {
  const line = lines[i].trim();
  if (line === SESSION_HEADER) {
    sessionStart = i;
  } else if (line === LESSONS_HEADER) {
    lessonsStart = i;
  } else if (sessionStart >= 0 && sessionEnd === -1 && line.startsWith('## ') && i > sessionStart) {
    sessionEnd = i;  // NEVER REACHED when Lessons header matched
  }
}
```

**Effect:** Session State section extended to end-of-file, swallowing stale
duplicate fields (e.g. old `Current Phase: build` dumped below Lessons by bug 2).

**Fix:** Changed to independent `if` statements so section-end detection runs
regardless of whether a header-match branch fired.

### Bug 2: appendLesson corrupts file when inserting new Lessons section

**File:** `cli/lib/progress.mjs`, `appendLesson()` function, no-Lessons-section branch

**Root cause:** When no `## Lessons` existed but a `## Session State` header
existed with field lines after it, code found the last `## ` header and spliced
the new Lessons section AFTER the header but BEFORE the content lines:

```javascript
// BROKEN: inserts after header, before its content
const lastHeader = lines.filter(...pop();
const after = lines.slice(lastHeader + 1);  // includes Session State fields!
newContent = before + '\n' + LESSONS_HEADER + '\n\n' + lessonLine + '\n' + after;
```

Result: `## Lessons` was placed between `## Session State` and its field lines,
dumping the fields below Lessons. Subsequent calls to writeSessionState produced
a file with two sets of session fields (one correct in SS section, one stale
below Lessons), and readProgress (due to bug 1) read the stale copy.

**Fix:** Simply append at end when no Lessons section exists.

- Template `templates/progress.md` fields match `DEFAULT_SESSION_FIELDS` in progress.mjs
- `learn.mjs` correctly imports progress.mjs and calls appendLesson
- `status.mjs` correctly imports progress.mjs and calls readLessons

- CLI entry point routes both learn and status commands

## Remaining Findings

- learn.mjs uses `typeof rawTarget === 'string'` guard but NOT `resolve()` for --target.
  Functionally correct (getProgressPath calls resolve internally) but inconsistent
  with status.mjs and the documented convention.
- Error JSON uses `{error, message, exitCode}` schema on stderr, not the standard
  `{command, status, message}` contract on stdout. This is intentional via formatError().
- writeSessionState replaces ALL 5 fields atomically — callers must include all
  fields they want to preserve.
- Empty string values fall back to defaults due to `||` operator in rendering.
