## Bugs Found & Fixed

### Bug #1 — set-mode autopilot lacks DEFINE+ guard

**Spec requirement:** PROJECT_PLAN.md line 693: "Must be in DEFINE phase or later"

**Problem:** `set-mode autopilot` accepted any currentPhase (including null/undefined from fresh init). Users could switch to autopilot before any specs were written.

**Fix:** Added guard in `cli/commands/set-mode.mjs` that checks `currentPhase >= 'define'` using `getPhaseOrder()` index comparison. Error: "Autopilot requires DEFINE phase or later (current: "${phase}")".

**Cascading test failures:** T9 E.8, T11 testSetMode, T11 testAutopilotPause all called `set-mode autopilot` on a fresh project (null currentPhase). Fixed by adding `config set currentPhase define` before the autopilot switch.

### Bug #2 — continuePipeline lacks pause re-check (T10 gap closed)

**Spec requirement:** PROJECT_PLAN.md line 700: "Pause/resume: harness-dev pause → outer loop stops after current phase gate"

**Problem:** When autopilot auto-advanced via continuePipeline, there was no pause re-check before transitioning to the next phase. If the user paused during phase execution, the auto-advance would ignore it. The `transitionPhase` function unconditionally clears `config.paused = false`, so even if paused was set, the transition would wipe it.

**T10 audit note:** This was identified as "defensible" in the T10 audit (line 264). For T12, it was fixed.

**Fix:** Added `config.paused` check in `cli/lib/ralph-outer.mjs` before the autopilot transition. If paused, returns `{status: 'paused'}` with message "Autopilot paused after {phase}. Run: harness-dev resume".

### Bug #3 — Stale test I.8 expects resume = not_implemented

**Problem:** T10 test I.8 asserted `resumeData.status === 'not_implemented'` because resume was a T12 stub during T10. T11/T12 implemented resume (calls `set(targetDir, 'paused', false)`), so the test became stale.

**Fix:** Updated `test-t10.mjs` I.8 to assert `status === 'ok'` and verify phase proceeds after resume.

## Remaining Design Gaps

### Gap #1 — Autopilot auto-advance only for feature-iterate phases

`phase.mjs` only calls `continuePipeline()` when `loopResult.status === 'complete'`. Deliverable-retry phases (define, plan, review, ship) always return `'instruction'`. This means the "After gate passes, autopilot will continue to {nextPhase}" message is aspirational — there's no wiring from `validate` → `continuePipeline`. True auto-advance after gate validation would require either:
- Modify `validate` command to trigger `continuePipeline` on pass
- Modify `phase.mjs` to handle 'instruction' status differently in autopilot
- Have agent call `phase <next>` explicitly

This gap was inherited from T10 and is not introduced by T12.
