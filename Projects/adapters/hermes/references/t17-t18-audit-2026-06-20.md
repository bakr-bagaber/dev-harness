## T17 — Worktree Management

### Commands

| Subcommand | Behavior | Examples |
|---|---|---|
| `create <name>` | `git worktree add ../feat-<name> -b feat/<name>` + full `harness-dev init` scaffold + harness-config.json with worktree parent metadata | `harness-dev worktree create auth` |
| `list` | Parses `git worktree list`, enriches each with harness phase/stack from config | `harness-dev worktree list [--json]` |
| `prune` | Runs `git worktree prune` to clean orphaned metadata | `harness-dev worktree prune` |
| `remove <name>` | Runs `git worktree remove`, supports `--force` flag for dirty worktrees, `--delete-branch` flag | `harness-dev worktree remove auth --force` |

### Design Decisions

- **Scaffold via `init`:** After creating the git worktree, `harness-dev init --stack <detected> --force --no-git --json` is called inside the worktree directory to produce a full 19-20 file scaffold (matching `init` behavior), then harness-config.json is overwritten with worktree-specific settings.
- **Parent tracking:** Each worktree's `harness-config.json` gets a `worktree: { parent, name, branch }` field so tools can trace back to the main repo.
- **Stack detection:** The stack is auto-detected from the (currently empty) worktree directory and falls back to `generic`. Users can override with `harness-dev init --stack <name> --force` inside the worktree.
- **No merge-on-remove:** Unlike the plan's optional merge-before-remove, we only remove the worktree tree and optionally delete the branch. Merge decisions belong to the user/git workflow.
- **scripts/clean-worktrees.sh:** Not implemented — git worktree prune handles orphaned metadata automatically.

## T18 — Rollback & Branch Recovery

### Commands

| Subcommand | Behavior | Examples |
|---|---|---|
| `rollback list` | Reads `phase/*`, `iter/*`, `manual/*` tags sorted by date; also lists `recovery/*` branches | `harness-dev rollback list [--json]` |
| `rollback to <tag>` | Stashes uncommitted changes, then `git checkout <tag> -- .` + explicitly restore harness state files | `harness-dev rollback to iter/3` |
| `rollback branch <tag>` | `git checkout -b recovery/from-<name> <tag>` | `harness-dev rollback branch iter/3` |
| `checkpoint create <label>` | `git tag -a manual/<label> -m "checkpoint: <label>"` with clean-tree guard | `harness-dev checkpoint create before-refactor` |

### Tag Naming Convention

| Prefix | Created By | Example |
|---|---|---|
| `phase/<name>` | Outer loop (when `git.autoTag` enabled) | `phase/define`, `phase/build` |
| `iter/<N>` | Inner loop (when `git.autoCommit` enabled) | `iter/1`, `iter/5` |
| `manual/<label>` | `checkpoint create <label>` | `manual/before-refactor` |
| `recovery/*` | `rollback branch <tag>` | `recovery/from-iter-3` |

### Design Decisions

- **rollback to vs rollback branch:** `rollback to` restores the working tree in-place (stays on current branch, files overwritten). `rollback branch` creates a new branch at the checkpoint — this is the "middle iteration" workflow: user picks a checkpoint, creates a branch, continues from there, and merges back if better.
- **Clean-tree guard:** `checkpoint create` refuses to create a tag if the working tree has uncommitted changes — ensures checkpoints are always semantically meaningful snapshots.
- **No auto-tagging in inner/outer loops yet:** The tags `phase/*` and `iter/*` are created by auto-tagging in T9 (outer loop) and T8 (inner loop) when `git.autoTag` is enabled. That auto-tagging code was already in the state machine from T9/T12. Manual checkpoints (`manual/*`) always work.
