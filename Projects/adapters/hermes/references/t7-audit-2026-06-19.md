## Bugs Found & Fixed

### BUG #1 (FIXED) — `checkTagged` only detects annotated tags

**File:** `cli/lib/gates.mjs:137-144`  
**Severity:** Medium  
**Root cause:** `git describe --exact-match HEAD` only finds annotated tags by default. The `checkpoint create` command (T12) creates lightweight tags, and typical git workflows default to `git tag <name>` (lightweight). This meant the SHIP gate would always fail for lightweight tags.

**Fix:** Added `--tags` flag:
```js
// Before:
'git describe --exact-match HEAD 2>/dev/null'
// After:
'git describe --exact-match --tags HEAD 2>/dev/null'
```

**Test:** C.8 (lightweight tag fail detection), C.9 (annotated tag pass)

---

### BUG #2 (FIXED) — SHIP gate uses `git diff --quiet` instead of `git status --porcelain`

**File:** `cli/lib/gates.mjs:88-95, 99-111, 207-211`  
**Severity:** Low (SHIP gate has no consumers yet)  
**Root cause:** The SHIP phase spec explicitly calls for `git status --porcelain` clean check (including detection of untracked files), but the implementation reused `checkGitCleanSimple` which only runs `git diff --quiet` (tracks changes to tracked files only, ignoring untracked files).

**Fix:** Created `checkGitStatusClean()` function using `git status --porcelain` and wired it to the SHIP phase check registry:
```js
// New function uses git status --porcelain:
const { out } = execCheck('git status --porcelain 2>/dev/null', targetDir);
const clean = out.length === 0;
```

**Tests:** C.8 (SHIP with untracked files correctly fails)

---

### BUG #3 (FIXED) — `validate` exits 0 even when checks fail

**File:** `cli/commands/validate.mjs` (both JSON and human output paths)  
**Severity:** Medium  
**Root cause:** The validate command produced output but never called `process.exit(EXIT.VALIDATION_FAILURE)` when checks failed. The help text documents "1 = Validation failure" but the code always returned exit 0. The JSON output path had an early `return` before the exit code check.

**Fix:** Added `process.exit(EXIT.VALIDATION_FAILURE)` in both the JSON output path (before the early return) and the human output path. The `process.exit()` calls `EXIT.VALIDATION_FAILURE` (value: 1) from `errors.mjs`.

**Tests:** CLI-4.4 (pass=0, fail=1 exit code verification)

**Status:** Not implemented in `validate.mjs`. The `help.mjs` advertises the flags, but `validate.mjs` only handles `--phase` and `--target`. When T8 (Inner Ralph Loop) is implemented, the CLI flags will be needed.

---

### GAP #2 — Missing per-phase checks vs spec table

The spec's check table (lines 355-364) lists many checks per phase. Current implementation is a pragmatic subset focused on git and stack-based commands:

| Phase | Spec says | Implemented | Coverage |
|-------|-----------|-------------|----------|
| INIT | git-repo, config-exists, feature_list.json valid, init.sh executable | git-repo, config-exists, init-executable | 3/4 |
| DEFINE | feature branch check | feature-branch | 1/1 |
| PLAN | plan.json valid, tasks ≤1 window, git-clean, DAG acyclic | git-clean | 1/4 |
| BUILD | git-clean, lint, imports resolve, tests pass, feature_list.json updated, progress.md appended | git-clean, lint, tests | 3/6 |
| VERIFY | git-clean, full suite, coverage ≥80%, browser evidence | git-clean, tests | 2/4 |
| SIMPLIFY | git-clean, no dead code, no unused imports, no commented-out code, nesting ≤4 | git-clean | 1/5 |
| REVIEW | branch up-to-date, evaluator approves, sprint contract met | branch-up-to-date | 1/3 |
| SHIP | status clean, tagged, rollback script exists, changelog updated | git-status-clean, tagged, changelog | 3/4 |

**Recommendation:** The current subset is appropriate for v0.2 — stack-agnostic checks (git, file existence) are universally applicable, while domain-specific checks (DAG acyclicity, coverage thresholds) require user configuration. Future iterations can add optional checks via the `gates.checks` array.

---

### GAP #3 — Spec footnote contradicts check table for INIT

**Spec:** Line 366 says "INIT is skipped for git checks (repo is being created)". But the INIT check table includes `git rev-parse --git-dir` (a git check). The implementation follows the table, not the footnote.

**Recommendation:** Update the footnote to clarify that the git-repo check validates that init *created* a git repo (an essential INIT goal), while other git checks like clean/tagged are what's "skipped" for INIT. Or accept the minor inconsistency.
