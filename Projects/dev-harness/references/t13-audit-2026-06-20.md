## Bugs Found & Fixed

### Bug #1 — `recentLessons` outputs strings instead of objects

**File:** `cli/commands/status.mjs:78`

**Problem:** `recentLessons: recentLessons.map(l => l.text)` returned an array of bare strings. The spec (PROJECT_PLAN.md lines 757-760) shows `recentLessons` as array of strings, but this loses metadata (date, author) that consumers need for rendering. The progress.mjs module stores lessons as objects with `{date, author, text}`.

**Symptoms:** Tests T6 J.4 and K.2 failed:
- J.4: `parsed.recentLessons[0].text` was `undefined` (because `recentLessons[0]` was a string, not an object)
- K.2: `typeof lesson.date === 'string'` was `false` (same reason)

**Fix:** Changed to `recentLessons: recentLessons.map(l => ({ date: l.date, author: l.author, text: l.text }))` — returns full lesson objects. This is more useful for API consumers while preserving all information.

**Impact:** Low — the JSON shape changed from `["text1", "text2"]` to `[{date, author, text}, ...]`. Only affects consumers of the `--json` output. Human output unchanged.

### Bug #2 — T5 CLI tests use wrong field names for status JSON

**Files:** `test-t5-cli.mjs` lines 88-92, 110-111

**Problem:** The T5 CLI integration tests checked for `r.phase` and `r.gates.enabled` in the status JSON output. The actual status.mjs output uses `currentPhase` (not `phase`) and `gateStatus` (not `gates.enabled`).

**Symptoms:** Tests CLI-1.2 and CLI-1.4 failed:
- `r.phase` was `undefined` (field is `currentPhase`)
- `r.gates.enabled` was `undefined` (field is `gateStatus` at top level, not nested)

**Fix:** Updated tests to check `r.currentPhase === null` and `r.gateStatus === 'disabled'`.

**Impact:** Low — these tests were failing since T13 was implemented. Fix brings them in line with the actual output contract.
