## Issues Found & Fixed

### BUG #1 (FIXED) — validateConfig reports `currentPhase: null` as missing

**File:** `cli/lib/state.mjs:387-396`
**Severity:** Medium
**Root cause:** `validateConfig()` treated `null` as missing for all required fields. But `currentPhase: null` is a valid value meaning "pipeline not started" (per JSON schema `type: ["string", "null"]`).

**Fix:** Added nullable field set:
```js
const nullable = new Set(['currentPhase']);
if (cfg[field] === undefined || (!nullable.has(field) && cfg[field] === null)) {
  missing.push(field);
}
```

**Status:** FIXED in this audit.

---

### ISSUE #2 (MINOR) — `git.hasUpstream` never updated in `transitionPhase`

**File:** `cli/lib/state.mjs:365-368`
**Severity:** Minor
**Description:** `transitionPhase` updates `git.branch`, `git.clean`, and `git.lastCommitMessage` but NOT `git.hasUpstream`. The default is `false` — it stays `false` even when the repo tracks an upstream branch. No consumer uses this field yet, so no production impact.

**Impact:** Low — field is present in schema and defaults but always `false`.

**Suggested fix:** Add upstream detection:
```js
config.git.hasUpstream = getGitUpstream(targetDir);
```

Where `getGitUpstream()` runs `git rev-parse --abbrev-ref --symbolic-full-name @{u} 2>/dev/null` and returns `true`/`false`.

**Status:** NOT FIXED (documented for future).

---

### ISSUE #3 (MINOR) — Missing key returns `undefined`, omitted from JSON output

**File:** `cli/lib/state.mjs:200-210` (`resolveKey`)
**Severity:** Minor
**Description:** When a dot-notation key doesn't exist, `resolveKey` returns `undefined`. `JSON.stringify({value: undefined})` omits the field entirely:

```
{"command":"config","subcommand":"get","key":"nonexistent.key","status":"ok","message":null}
```
No `value` field present. For explicit `null`, it would show `"value": null`.

**Impact:** Low — consumers see status 'ok' and can infer the key doesn't exist from the missing `value` field, but it's ambiguous (is it missing or was the field not included in the response?).

**Suggested fix:** Change `resolveKey` to return `null` for missing keys instead of `undefined`, or add a `found` boolean.

**Status:** NOT FIXED (documented for consideration).

---

### ISSUE #4 (DESIGN) — `init` not in default `phases.enabled`

**File:** `cli/lib/state.mjs:20-29`, Default config line 93
**Severity:** Not a bug — design choice
**Description:** `PHASE_ORDER` includes `init` but the default config's `phases.enabled` is `['define', 'plan', 'build', 'verify', 'review', 'ship']`. This means `harness-dev phase init` is never valid by default.

**Rationale:** The `init` phase is handled by the `harness-dev init` CLI command (creates scaffold + config). The phase pipeline starts at `define`. This is consistent with the T5 architecture doc.

**Recommendation:** Document this distinction in skill references and the AGENTS.md template.

---

### ISSUE #5 (CODE QUALITY) — Empty if block in `set()`

**File:** `cli/lib/state.mjs:255-258`
**Severity:** Cosmetic
**Description:** The `set()` function has an empty if-block with only a comment. Logic works correctly — the comment documents the intentional use of defaults when no config exists. Could be simplified to:

```js
export function set(targetDir, key, value) {
  const { config } = loadConfig(targetDir); // config is always filled (file or defaults)
  setKey(config, key, value);
  return saveConfig(targetDir, config);
}
```

**Status:** NOT FIXED (cosmetic).
