## Bugs Found & Fixed

### Bug #1 ‚Äî `getPhaseOrder([])` returns default instead of empty array

**File:** `cli/lib/state.mjs:37-43`

**Problem:** When `phases.enabled` was explicitly `[]`, `getPhaseOrder` treated it as falsy (`enabled && enabled.length > 0` was false for `[]`) and returned the default list (all phases except simplify). This meant disabling all phases was impossible ‚Äî the empty-enabled configuration was silently ignored.

**Symptoms:** `runAutopilot` with no enabled phases returned `ok: true` instead of an error. The `phases.enabled: []` override was functionally a no-op.

**Fix:** Changed the guard to `enabled === undefined || enabled === null` for the default fallback, with a separate `Array.isArray(enabled)` check for the explicit list path.

**Verification:** `getPhaseOrder([])` now returns `[]`. All other call sites behave identically because they pass `undefined` (from `config.phases?.enabled` when the field doesn't exist), which correctly falls through to the default.

### Bug #2 ‚Äî `isValidTransition` rejects same-phase transitions

**File:** `cli/lib/state.mjs:56-63`

**Problem:** `isValidTransition('build', 'build')` returned `false` because `toIdx === fromIdx + 1` is false when fromIdx === toIdx. This meant re-running `harness-dev phase build` when already in the `build` phase was impossible ‚Äî the command would error with "Invalid transition: build ‚Üí build."

**Rationale for fix:** Same-phase transitions are needed for:
- Re-running a phase after fixing issues
- Agent workflow: validate ‚Üí fix ‚Üí re-run phase
- Allowing explicit retries without state manipulation

**Fix:** Added `if (fromPhase === toPhase) { return true; }` before the sequential check.

**Side effect:** `transitionPhase` was also updated to skip recording a gate history entry (`recordGate`) when transitioning to the same phase ‚Äî recording a "pass" event for a re-run was misleading.

### Identified Gaps

#### GAP #1 (CRITICAL) ‚Äî No retry counting / iteration tracking

**Spec says** (lines 420-470): The algorithm describes retry counting with `‚â§ maxRetries` and `> maxRetries ‚Üí escalate to human`. T9 has no mechanism for this.

**Reality:** `runPhase()` always returns `iteration: 1`. `continuePipeline` does not track iterations or retries. There is no state persistence for iteration counts. The `maxRetries` config value is read by `ralph-inner.mjs` but only passed through to the instruction text ‚Äî never enforced.

**Impact:** If an agent fails to produce a passing result, the outer loop does not detect the failure cycle. It continues to the next phase (or in copilot mode, prints the next instruction). The only way to trigger a retry is manual ‚Äî the agent re-running `harness-dev phase <name>`.

**Fix requires:** A commit-style approach where each phase execution is recorded with its iteration count in `harness-config.json`. Before running a phase, check the count against `maxRetries` and return `status: 'escalated'` if exceeded.

#### GAP #2 (MINOR) ‚Äî `paused` flag not checked by outer loop

**Spec says:** `harness-dev pause` should stop autopilot after the current phase gate.

**Reality:** `continuePipeline` and `runAutopilot` do not check `config.paused`. If autopilot mode is active, pause has no effect ‚Äî the pipeline keeps advancing.

**Fix requires:** Adding `if (config.paused) { return { ok: true, status: 'paused', ... }; }` at the start of `continuePipeline`.

#### GAP #3 (MINOR) ‚Äî No git auto-tag on pipeline iteration

**Spec says** (line 526): "Tag iteration (if git.autoTag is true)"

**Reality:** Neither T8 nor T9 implements git auto-tagging. The `git.autoTag` config value is never read.

#### GAP #4 (MINOR) ‚Äî No auto-append to progress.md

**Spec says** (line 527): "Append lessons to progress.md"

**Reality:** Left to the agent via `harness-dev learn`. T9 does not interact with progress.mjs.

#### GAP #5 (EXISTING DEFERRAL) ‚Äî `--feature`/`--task` on validate.mjs

**Since T7 audit:** The flags are now parsed by validate.mjs and accepted by gates.mjs, but per-feature/task filtering is still unimplemented. gates.mjs always runs full phase-level checks regardless of the feature/task arguments.

## Verdict

**T9 is ready for follow-on tasks (T10: Phase Orchestrator, T11: Copilot Mode, T12: Autopilot Mode) with the following recommendations:**

1. **MUST FIX:** Add retry counting before shipping autopilot mode. Without it, the outer loop cannot detect escalation conditions, and a failing phase will silently advance the pipeline. This is the single most important gap.

2. **SHOULD FIX:** Add `paused` flag check to `continuePipeline` ‚Äî otherwise `harness-dev pause` is a no-op in autopilot mode.

3. **DEFER:** Git auto-tag, progress.md auto-append, and per-feature/task gates ‚Äî these are quality-of-life features that don't block pipeline correctness.

The core T9 implementation (`ralph-outer.mjs`) is structurally sound. The 51-pass test suite covers all common paths, all edge cases, and all spec gap documentation. The two bugs discovered in `state.mjs` (empty enabled list, same-phase transitions) were fixed with zero regression.

**T9 Verdict: READY WITH DEFERRED ESCALATION** (gate verb: implement retry counting in outer loop)
